{% comment %} CommencementProvenance detail — the *why* behind an in-force date. `commencement` is one entry dict (RegulationClause.commencement / Regulation.commencement, mirroring CCM's CommencementProvenance): the verbatim commencement subsection, how the date was established (`source`), and for derived dates the `depends_on` statute + `computation`. Rendered as the body of a popup on both the regulation clause page and the provision page, so it carries no modal chrome of its own. `effective_date` (optional) is the real date object the host surface already holds (clause/version effective_date) — preferred for display since the entry's own date is a raw ISO string from JSON. `relation` (optional) reframes the whole popup. Default/"start": this entry is *why this version came into force* (labels read "Commencement" / "In force"). "end": the entry is the SUCCESSOR's commencement — the event that ended the version being viewed — so it gains a header strip naming the subject change and relabels the date "Next version in force" (the same instant this version ceased). The body is identical either way; only the framing differs. `next_version_url` (optional, end framing only) is that successor version's permalink, and doubles as the end-context discriminator. PRESENT → context A: the version was superseded by the next version of this same provision; the "Next version" references link to it. ABSENT → context B: this was the edition-final version, replaced by the next EDITION's base regulation (no same-provision successor to link), so the wording switches to "next edition". The two contexts are the only ones the End modal opens in (a live version with no successor yields no end date, so the trigger never shows). {% endcomment %}
