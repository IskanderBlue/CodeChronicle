# 3.3 — Transition Banner and Compare Cards

## Status: implemented, pending browser verification

## What

Render overlapping old/new edition results inside one comparison-capable result
card with a transition banner.

## Why

Without UI merging, overlap-aware search would surface duplicate-looking results
that do not explain their relationship.

## Changes

- Merge paired transition results for the same logical section/table into one
  grouped compare result.
- Render a transition banner with:
  - queried date
  - new version effective date
  - old version last overlap/applicability date
  - transition type
  - applicability statement
  - citation text
- On desktop, show side-by-side panes.
- Emphasize the edition that most directly matches the queried date.
- Each pane uses its own result span fields:
  - `page`
  - `page_end`
  - `initial_page_top`
  - `final_page_bottom`

## Completion Conditions

- Transition-related duplicate results are merged into one compare card.
- Banner content explains why both editions are shown.
- Query-matching edition has stronger emphasis than comparison edition.
- Each pane can render its own PDF span independently.

## How to Test

### Unit test: `test_formatter_merges_transition_pair_into_single_compare_result`

Assertions:

```python
assert len(formatted_results) == 1
assert formatted_results[0]["result_type"] == "transition_compare"
assert len(formatted_results[0]["versions"]) == 2
```

### Template test: `test_transition_compare_card_renders_banner_fields`

Assertions:

```python
assert 'grace period' in rendered.lower()
assert 'Queried date' in rendered
assert 'Citation' in rendered
```

### Browser/manual test: `transition_compare_card_shows_two_independent_pdf_spans`

Expected:

- Left/right panes both render.
- Each pane starts at its own first page and span bounds.
- Primary pane is visually emphasized.
