# 6.1 — Engagement / Click Tracking

## Status: implemented, pending browser verification

## What

Add server-side event tracking for what users *do with* search results —
which provision versions, regulation clauses, and result links they open —
to complement the search-input tracking we already capture in
`SearchHistory`.

## Why

Today we record every search's inputs (`SearchHistory.parsed_params` holds
keywords, date, province, building_type) and `result_count`/`top_results`,
but we have **no record of engagement**: no model logs that a user opened a
particular `CodeEditionProvisionVersion`, drilled into a `RegulationClause`,
or followed a link out of a results list. The detail views
(`core/views/regulation.py:regulation_detail`, `provision_permalink`) just
render templates. Without this we can measure demand (what people look for)
but not value delivered (what they actually read), and we can't compute
click-through rate per query.

## Design

### New model: `EngagementEvent` (core)

A single thin, append-only event table — one row per tracked interaction.

| Field | Type | Purpose |
|---|---|---|
| `user` | FK→User, null=True | who (authed) |
| `ip_address` | GenericIPAddressField, null=True | who (anonymous) |
| `event_type` | CharField(choices) | `provision_version_view`, `regulation_clause_view`, `result_link_click` |
| `object_type` | CharField | model label of the target, e.g. `CodeEditionProvisionVersion` |
| `object_id` | BigIntegerField, null=True | target pk (kept loose — not a hard FK, see Notes) |
| `search` | FK→SearchHistory, null=True | the search this engagement came from, when known |
| `context` | JSONField(default=dict) | free-form (provision_id, division, reg_id, clause_id, source surface) |
| `timestamp` | DateTimeField(auto_now_add) | when |

Indexes: `(event_type, timestamp)`, `(object_type, object_id)`,
`(search,)`. `db_table = "engagement_events"`.

Rationale for a single generic table over per-target tables: the event
shapes are near-identical and we want cheap "top N most-viewed versions /
clauses" aggregates without UNIONing. Mirrors how `SearchHistory` already
carries heterogeneous detail in a JSONField.

### Capture paths

Two complementary mechanisms:

1. **Server-rendered view hits** — in `regulation_detail`,
   `provision_permalink`, and the provision-version render path, write an
   `EngagementEvent` (non-fatal `try/except`, matching the
   `search_service.py:74` pattern). This catches direct/permalink navigation
   with zero client JS.

2. **Result-link clicks** — a lightweight beacon endpoint
   `POST /api/event` (Django Ninja, in `api/`). The results templates are
   already HTMX-driven, so wire result links with an `hx-post` (or a small
   `fetch` on click) carrying `{event_type, object_type, object_id,
   search_id, context}`. The `search_id` ties the click back to the
   originating `SearchHistory` row so click-through rate per query is
   computable. Keep the search id in the rendered results so the beacon can
   echo it back.

### Privacy / rate-limit interaction

- Reuse the existing anonymous-vs-authed identity split (`user` else
  `ip_address`) already used by `SearchHistory` and `RateLimitMiddleware`.
- The beacon endpoint must NOT count against search rate limits — it is not
  a search. Confirm `RateLimitMiddleware` only gates the search path; add an
  exemption if it gates by path prefix.

## Changes

- `core/models.py`: add `EngagementEvent` + migration.
- `core/views/regulation.py`: record view events in `regulation_detail` and
  `provision_permalink` (non-fatal).
- Provision-version render path (search/viewer): record version-view events.
- `api/`: add `POST /api/event` beacon endpoint (validate payload, write
  event, return `{"success": true}` envelope per CLAUDE.md API contract).
- Results/detail templates: emit `data-*`/`hx-post` so links fire the beacon;
  thread the `search_id` through to results so clicks attribute to a search.
- `core/admin.py`: register `EngagementEvent` read-only for inspection.

## Completion Conditions

- Opening a provision version, regulation detail, or provision permalink
  writes exactly one `EngagementEvent` of the right `event_type`.
- Clicking a result link fires the beacon and writes one event carrying the
  originating `search_id`.
- Event writes are non-fatal: a tracking failure never breaks the page or
  the search.
- Beacon endpoint is exempt from search rate limiting.
- Anonymous and authenticated events both record correctly (ip vs user).

## How to Test

### `engagement_event_written_on_provision_version_view`
- Request a provision permalink → assert one `EngagementEvent` with
  `event_type="provision_version_view"` and matching `object_id`.

### `engagement_event_written_on_regulation_detail`
- GET `regulation_detail` → assert a `regulation_clause_view` /
  `regulation_view` event row.

### `beacon_endpoint_records_result_click_with_search_link`
- POST `/api/event` with a valid payload including `search_id` → 200,
  `{"success": true}`, and an `EngagementEvent` linked to that
  `SearchHistory`.

### `tracking_failure_is_non_fatal`
- Force the event write to raise (patch `.objects.create`) → the page/search
  still returns 200.

### `beacon_not_rate_limited`
- Exhaust the search rate limit, then POST `/api/event` → still succeeds.

## Depends On

- `SearchHistory` (exists) — for the `search` FK and click-through joins.
- The search results render path that exposes `top_results` / result ids —
  needs the `SearchHistory` id surfaced into the results template.

## Notes

- `object_id` is intentionally a loose `BigIntegerField`, not a hard FK:
  events outlive the rows they point at (re-loads via `load_edition` replace
  provision/version pks), and an append-only analytics log should not cascade
  or block deletes. Resolve targets at report time, tolerating misses.
- Out of scope for this task: dashboards/reporting UI, GA/Plausible/PostHog
  or any third-party client analytics, and aggregation materialized views.
  This task only lands the capture layer. A follow-up (`6.2`) can add
  reporting (e.g. top-viewed provisions, CTR per keyword) once data exists.
- Follow the repo terminology: events about generic structural units use
  "provision", not "section".

## Implementation Update (2026-06-02)

Landed the capture layer. Files:

- `core/models.py`: `EngagementEvent` model (`engagement_events` table) +
  migration `0023_engagementevent.py`. `search_id` FK id-shadow declared for
  Pyright, matching the repo convention.
- `core/events.py` (new): `record_event(request, …)` — best-effort, swallows
  failures, derives user-vs-IP identity the way the search path does, and
  coerces a string `search_id` (drops it if not a positive int).
- `core/views/regulation.py`: `regulation_detail` records `regulation_view`;
  `provision_permalink` records `provision_version_view` pinned to the linked
  version.
- `core/views/search.py`: `viewer_section_content` records a
  `provision_version_view` for the matched provision's in-force version — this
  is the **primary search-result click-through** capture, server-side, no
  client beacon needed.
- `services/search_service.py` → `search.py` view → `search_results_partial.html`
  → `search.html`: threads `search_history_id` so the viewer drill-in
  attributes back to its originating search (stashed in `#cc-search-id`, read
  by `sectionContentUrl()`).
- `api/views.py`: `POST /api/event` beacon (public, paid-access guard
  deliberately skipped) for out-of-band clicks (external source links, PDF
  downloads). Validates `event_type` against the model's choices.
- `core/admin.py`: read-only, add-disabled `EngagementEventAdmin`.
- `core/tests/test_engagement.py`: 8 tests — all green.

### Deviations from plan

- Used `regulation_view` (not `regulation_clause_view`): the detail page is a
  whole regulation, not a single clause.
- The main result→provision click-through is captured **server-side** in the
  existing `viewer_section_content` round-trip rather than via a client
  beacon. More robust and directly testable; the `/api/event` beacon remains
  for clicks with no server round-trip.

## Beacon Wiring Update (2026-06-02)

Wired the `/api/event` beacon to the two in-app links that leave without a
server round-trip, both in the search viewer overlay (`templates/search.html`):

- **Source link** (citation → e-Laws/source_url): Alpine `@click="recordLinkClick('source_link')"`.
- **PDF download** (`pdf_download_url`): the anchor lives inside a JS-cloned
  `<template data-pdf-picker-template>`, outside Alpine's reactive DOM, so
  `@click` can't bind. Marked it `data-beacon-pdf` and added a delegated
  document click listener that routes through the overlay component's
  `recordLinkClick()` via the existing `withViewerOverlay()` helper.
- New `recordLinkClick(surface)` method POSTs `result_link_click` to
  `/api/event` with `keepalive: true` (survives the new-tab navigation),
  `credentials: 'same-origin'` (session cookie → user attribution), and the
  stashed `#cc-search-id` for click-through attribution. Wrapped + `.catch()`
  so a tracking failure never blocks the click.

Gotcha avoided: the inline `{# … #}` note on the PDF link is kept on ONE line
— a multi-line Django comment tag leaks as literal page text (project memory).

## Gap-Closing Update (2026-06-02)

Live testing revealed the real gap: the primary "open a provision" action is
**expanding the result accordion inline in the results list**, which is a pure
Alpine show/hide with **no server round-trip** — so nothing was recorded. (The
viewer-overlay path that *does* hit the server is a separate, less-used action;
and this edition has no PDFs, so the PDF early-return theory was moot.)

- **Inline result-expand now beacons** (`window.ccRecordView` →
  `provision_version_view`, `surface: results_expand`, `search_id` attached).
  Wired via an `open()`/`toggleChild()` method on the results-partial Alpine
  `x-data`; children record only on expand (not collapse). `object_id` is null
  — the client doesn't know the version pk — with `code`/`provision_id`/
  `division` in context for report-time resolution.
- **Shared poster** `window.ccPostEvent` in `search.html` (auto-attaches the
  stashed `#cc-search-id`); the viewer's `recordLinkClick` now uses it too.
- **Gap #1 closed** via `decorateResultLinks()`: a post-render pass over
  `#results-area` that appends `?search_id=` to `/provision/…` and
  `/regulation/…` links, so permalink/clause click-throughs attribute to the
  search through the existing server-side capture (no double count).

Gotcha hit & recorded to memory: a `{% url %}` written **inside a JS comment**
in a `.html` template still gets compiled by Django (text-based parser) →
argument error, page 500. Reworded the comment.
