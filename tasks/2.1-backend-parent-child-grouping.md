# 2.1 — Backend Parent/Child Grouping

## What

Add a formatter-layer grouping step that can collapse dense sets of sibling hits
 into a single grouped result.

## Why

Search should still retrieve specific nodes, but the UI should not overwhelm the
user with long runs of sibling results when nearly an entire parent scope has
matched.

## Changes

- Add a grouping stage after search retrieval and before final display shaping.
- Keep retrieval at article/subsection/table granularity.
- Grouping must use node hierarchy coverage, not PDF geometry.
- Group when more than 80% of a parent's direct child nodes are present in the
  result set.
- Do not group when only one child matches.
- Do not group when the returned child coverage is 80% or less.
- Preserve enough metadata for the frontend to know:
  - parent node id/title
  - ordered child nodes
  - which child nodes actually matched
  - top-scoring child

## Completion Conditions

- Grouping logic uses direct child node counts, not page counts.
- Returned grouped results are deterministic for the same input order/data.
- Non-grouped standalone results still pass through unchanged.
- The output shape is explicit enough for the frontend task to render grouped
  cards without re-querying hierarchy.

## How to Test

### Unit test: `test_group_results_collapses_when_more_than_80_percent_of_direct_children_match`

Setup:

- Parent `B-3.2.9`
- Direct children: `B-3.2.9.1`, `B-3.2.9.2`, `B-3.2.9.3`, `B-3.2.9.4`, `B-3.2.9.5`
- Returned hits: first 5 of 5 or 5 of 6 equivalent coverage > 80%

Assertions:

```python
assert len(grouped_results) == 1
assert grouped_results[0]["group_type"] == "parent_children"
assert grouped_results[0]["parent_id"] == "B-3.2.9"
assert len(grouped_results[0]["children"]) == 5
```

### Unit test: `test_group_results_does_not_group_at_or_below_80_percent`

Setup:

- 4 direct children total
- 3 matching children returned (75%)

Assertions:

```python
assert all(item.get("group_type") != "parent_children" for item in grouped_results)
assert len(grouped_results) == 3
```

### Unit test: `test_group_results_uses_children_not_pages`

Setup:

- One child spans 3 pages, another spans 1 page

Assertions:

```python
assert grouping_decision is based on child count only
```

Practical assertion form:

```python
assert grouped_results[0]["child_match_count"] == 2
assert grouped_results[0]["child_total_count"] == 2
```

### Unit test: `test_group_results_keeps_single_child_match_standalone`

Assertions:

```python
assert len(grouped_results) == 1
assert grouped_results[0].get("group_type") is None
```

## Depends On

- Existing `parent_id` data in imported nodes.

## Notes

- This task is about grouping semantics only. It should not alter PDF span
  rendering fields.
