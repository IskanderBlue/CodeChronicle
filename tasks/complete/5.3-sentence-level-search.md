# 5.3 Sentence-Level Search Support

## Problem

`SECTION_REF_RE` in `api/llm_parser.py` captures dotted article IDs (e.g. `3.1.6.4`) but not sentence-level subclauses like `3.1.6.4.(4)`. Users searching for a specific sentence get no `section_references` match — the subclause portion is silently dropped.

Current regex:
```python
SECTION_REF_RE = re.compile(
    r"\b((?:(?:table|[a-z])-)?\d{1,2}(?:\.\d{1,2}){1,4})\b",
    re.IGNORECASE,
)
```

## New ID Formats (from CodeChronicleMapping)

Appendix commentary headings that cover multiple sentences now use compound IDs instead of being expanded into duplicate entries:

| PDF text | Old ID(s) | New ID |
|---|---|---|
| `A-3.1.6.4.(3) to (7) Title` | `A-3.1.6.4.(3)`, `(4)`, `(5)`, `(6)`, `(7)` | `A-3.1.6.4.(3-7)` |
| `A-3.1.6.9.(1) and (2) Title` | `A-3.1.6.9.(1)`, `A-3.1.6.9.(2)` | `A-3.1.6.9.(1,2)` |
| `A-3.8.4.4.(2) and (4) Title` | `A-3.8.4.4.(2)`, `A-3.8.4.4.(4)` | `A-3.8.4.4.(2,4)` |

## What to Capture

Update `SECTION_REF_RE` to also match:

- Single subclauses: `3.1.6.4.(4)`
- Ranges: `3.1.6.4.(3-7)`
- Lists: `3.1.6.9.(1,2)` or `3.8.4.4.(2,4)`

Suggested pattern (append optional subclause group):
```
(?:\((\d+(?:-\d+|(?:,\d+)*))\))?
```

## Impact

- `extract_section_references` would return the full sentence ID when present
- `_search_code_db` already uses `icontains` so article-level searches are unaffected
- Scoring in `engine.py` (line 156) gives higher score for exact suffix match — sentence-level refs would benefit from this
