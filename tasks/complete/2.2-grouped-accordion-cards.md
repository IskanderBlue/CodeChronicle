# 2.2 — Grouped Accordion Cards

## Status: implemented, pending browser verification

## What

Render backend-grouped parent/child results as a single accordion card with an
ordered child list inside it.

## Why

Once grouping exists, the frontend needs a readable representation that shows
the relationship between matched children and filled-in context children.

## Changes

- Update result card rendering to support grouped-result objects.
- Grouped card header should show the parent identity.
- Child list should be ordered by code order, not score order.
- Visually distinguish:
  - top-scoring child
  - matched children
  - non-matching context children
- Expanding a grouped card should still expose the active PDF/HTML content for
  the selected child or default child chosen by the implementation.

## Completion Conditions

- Grouped results render as one accordion item, not many duplicate siblings.
- Child ordering is stable and code-order based.
- The highest-scoring child is visually distinct.
- Non-matching siblings are visibly lower emphasis but still readable.

## How to Test

### Template test: `test_grouped_result_renders_parent_header_and_children`

Assertions:

```python
assert 'B-3.2.9' in rendered
assert 'B-3.2.9.1' in rendered
assert 'B-3.2.9.2' in rendered
```

### Template test: `test_grouped_result_marks_top_scoring_child`

Assertions:

```python
assert 'data-top-scoring-child="B-3.2.9.2"' in rendered
```

### Browser/manual test: `grouped_card_displays_context_children_with_low_emphasis`

Expected:

- Matched children look primary.
- Filled-in context children are visibly subdued.

### Browser/manual test: `grouped_card_expands_without_duplicate_pdf_blocks`

Expected:

- One grouped accordion card appears.
- Expanding it does not duplicate the same PDF block for each child row.
