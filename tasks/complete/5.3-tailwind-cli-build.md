# 5.3 — Tailwind CLI Production Build

## Status: planned

## What

Replace the Tailwind CDN `<script>` tag with a pre-built CSS file generated by the standalone Tailwind CLI during CI.

## Why

The CDN version ships the full Tailwind framework and JIT-compiles in the browser. A CLI build produces a minimal CSS file with only the classes used in templates — smaller payload, no FOUC, no console warning.

## Approach

1. Add a `tailwind.config.js` with `content` pointing at `templates/**/*.html`.
2. Create `static/css/input.css` with `@tailwind base; @tailwind components; @tailwind utilities;` and any custom directives.
3. In `publish.yml`, cache and run the standalone Tailwind CLI binary before the Docker build:
   - Cache `tailwindcss-linux-x64` keyed on config hash so it only downloads once.
   - Run `./tailwindcss-linux-x64 -i static/css/input.css -o static/css/tailwind.css --minify`.
4. In `templates/base.html`, swap the CDN `<script>` for `<link href="{% static 'css/tailwind.css' %}" rel="stylesheet">`, gated on `DEBUG` so local dev can keep using the CDN.
5. Locally, optionally run `tailwindcss --watch` for the same live-rebuild experience.

## Acceptance

- Production pages load a single minified CSS file, no CDN script tag.
- Local dev still works with the CDN (no build step required).
- No console warning about CDN usage in production.
