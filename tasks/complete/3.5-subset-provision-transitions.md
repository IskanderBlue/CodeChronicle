# 3.5 — Subset (Provision-Scoped) Transitions

## Status: planned

## What

Extend the transition model to handle cases where only specific provisions of the
old edition remain valid during a transition window, rather than the whole code.

The key design principle: **tag the NEW edition's sections** with transition
metadata pointing back to the old edition's provisions. When a user sees a result
from the new edition, the annotation tells them which old-edition provisions are
still valid and until when.

## Why

OBC eLaws transition rules distinguish between "whole_code" and "provisions"
scope. A whole_code transition (e.g., OBC 2012 subsection 4.1.6) means the
entire old edition can still apply for in-stream projects. A provisions transition
(e.g., subsection 4.1.3) means only specific listed provisions — like
"Sentence 8.6.2.2.(5)" or "Item 329 of Table 1.3.1.2." — get the grace period.

With the current edition-scoped model (3.1 Tier 1), there is no way to
distinguish these cases or direct users to the specific old-edition provisions
that remain valid.

## Design

### Annotation on the new edition, not the old

When the new edition's section 8.6.2.2. appears in search results during a
provisions-scoped transition window, the result should carry metadata like:

```
"This section was amended on 2017-01-01. Sentence 8.6.2.2.(5) as it read on
 2016-12-31 may still apply to in-stream projects until 2017-07-01."
```

This avoids dual-version search complexity. The user sees the current (new)
edition result, with an annotation that the old version's specific provision
is still valid.

### Schema extension

Add optional Tier 2 fields to transition records in `transitions.json`:

```json
{
  "old_edition": "OBC_2012_v08",
  "new_edition": "OBC_2012_v09",
  "overlap_start": "2017-01-01",
  "overlap_end": "2017-07-01",
  "transition_type": "in_stream_project",
  "applicability_text": "...",
  "citation_text": "O. Reg. 332/12, s. 4.1.3",
  "scope": "provisions",
  "provisions": [
    {
      "new_section_id": "8.6.2.2",
      "old_provision_ref": "Sentence 8.6.2.2.(5)",
      "as_read_on": "2016-12-31"
    },
    {
      "new_section_id": "1.3.1.2",
      "old_provision_ref": "Item 329 of Table 1.3.1.2.",
      "as_read_on": "2016-12-31"
    }
  ]
}
```

Whole-code transitions omit `scope` and `provisions` (or set `scope` to
`"whole_code"`), preserving backward compatibility with the Tier 1 schema.

### Matching logic

`provisions[].new_section_id` is the key that links a transition annotation to
a search result. When a search result's `id` matches (or is a parent/child of)
a `new_section_id`, the transition annotation attaches to that result.

This is a prefix match: a result for section `8.6.2.` would pick up annotations
for `new_section_id: "8.6.2.2"` since it's a child. A result for `8.6.2.2`
matches exactly.

### Display

The annotation surfaces alongside the new-edition result, not as a separate
search result. Possible UI treatments:

- Inline note beneath the result title
- Expandable "transition context" slot (builds on 1.4 justification slot)
- Link/reference to the old edition's viewer page for that provision

## Changes

- Add optional `scope` and `provisions` fields to the transition schema.
- Update `load_transitions()` validation to accept the new fields.
- Update `get_active_transitions()` or add a new helper to return
  provision-level annotations matched against result section IDs.
- Add OBC eLaws provision-scoped transition records to `transitions.json`.
- Wire annotations into search result enrichment (extends 3.2 orchestration).

## Completion Conditions

- Whole-code transitions continue to work unchanged.
- Provision-scoped transitions attach annotations to matching new-edition results.
- A search hitting OBC 2012 section 8.6.2.2. during the 4.1.3 overlap window
  surfaces the old-provision annotation.
- A search hitting an unaffected section during the same window shows no
  annotation.

## How to Test

### Unit test: `test_provision_scoped_transition_loads_with_optional_fields`

```python
records = load_transitions()
prov_records = [r for r in records if r.get("scope") == "provisions"]
assert prov_records
assert "provisions" in prov_records[0]
assert prov_records[0]["provisions"][0]["new_section_id"]
```

### Unit test: `test_provision_annotation_matches_result_section_id`

```python
# Given a search result with id="8.6.2.2" during the overlap window,
# the annotation for Sentence 8.6.2.2.(5) should attach.
annotations = get_provision_annotations("OBC_2012_v09", "8.6.2.2", date(2017, 3, 15))
assert len(annotations) == 1
assert "8.6.2.2.(5)" in annotations[0]["old_provision_ref"]
```

### Unit test: `test_unaffected_section_gets_no_provision_annotation`

```python
annotations = get_provision_annotations("OBC_2012_v09", "3.1.1.1", date(2017, 3, 15))
assert annotations == []
```

## Notes

- The `old_provision_ref` uses the OBC's own citation style (Sentence, Article,
  Item, Subsection) which is finer-grained than the map file section IDs. The
  `new_section_id` bridges between the map's article-level granularity and the
  OBC's sentence-level references.
- Whole-code transitions (4.1.2, 4.1.4–4.1.7 in OBC 2012) do not need provision
  annotations — they use the existing Tier 1 edition-scoped model from 3.1.
- This task does not add the whole-code OBC transitions to `transitions.json`;
  that is a data curation step that can proceed independently under 3.4.
